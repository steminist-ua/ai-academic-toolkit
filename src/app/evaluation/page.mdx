# 8. Принципи перевірки якості й оцінювання

Оцінюється **не здатність згенерувати контент**, а глибина його осмислення та якість інтеграції. Ключові критерії:

- **Додана вартість:** обґрунтованість архітектурних рішень, оптимізації, оригінальність підходу.
- **Глибина розуміння:** здатність пояснити будь-яку частину роботи під час захисту (усний/live‑coding).
- **Інженерна культура:** якість коду, документації, тестів та відтворюваність результатів.
- **Критична взаємодія з ШІ:** якість та повнота рефлексії, описаної у Декларації.
- **Комунікація/документація** — ясність README, діаграми, коментарі.
- **Репозиторій як журнал навчання:** `prompts.md`, `decisions/ADR-*.md`, `CHANGELOG.md`, автогенеровані артефакти помічені тегом `generated`.
- **Code review і перевірка авторства:** короткі інтерв’ю, випадкові запитання по коду, live‑coding зміни; аналіз стилю коду (стилометрія) **не є самостійною підставою** для висновку про недоброчесність, але може слугувати приводом для додаткової співбесіди.
- **Тестування:** вимога до unit/integration/prop‑based tests; окремі тести на крайові випадки, на які LLM часто «помиляється».
- **Щоденники часу й внеску:** для команд — матриця ролей, `CONTRIBUTORS.md`, автоматичний облік PR/комітів.
- **Режими іспитів:** за замовчуванням **без** ШІ. За потреби — контрольоване середовище (локальний офлайн‑асистент із логуванням; обмежений контекст; без доступу до зовнішньої мережі).
    - Режим Заборона — жодних ШІ.
    - Режим Пісочниця — локальний інструмент без інтернету, лог промптів, ліміт 5 звернень/год.
    - Режим Офлайн-асистент — заздалегідь підготовлений набір підказок/довідок; звернення фіксуються.
