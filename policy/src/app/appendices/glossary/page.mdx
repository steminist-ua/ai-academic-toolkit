# Додаток Ж. Глосарій термінів

| Термін | Пояснення |
| --- | --- |
| **A11y (Accessibility)** | Скорочення від «Accessibility»; узагальнена назва практик і вимог доступності цифрових продуктів (див. також WCAG). |
| **A11y.md** | Короткий файл/нотатки з доступності у репозиторії (чек-листи, контраст, alt‑тексти, навігація з клавіатури). |
| **Acceptance criteria** | Критерії, за якими визначається, чи функція відповідає вимогам. |
| **Accessibility (WCAG)** | Доступність цифрових продуктів для людей із різними можливостями; WCAG — міжнародний стандарт Web Content Accessibility Guidelines. |
| **ADR (Architecture Decision Record)** | Журнал архітектурних рішень, документ, що фіксує ключові технічні рішення та їхнє обґрунтування. |
| **AI code assistant (код-асистент)** | Інструмент ШІ для автодоповнення, рефакторингу та пояснення коду (наприклад, GitHub Copilot). |
| **API (Application Programming Interface)** | Програмний інтерфейс застосунку, набір правил і методів взаємодії між програмами. |
| **ARIA (Accessible Rich Internet Applications)** | Набір атрибутів для покращення доступності веб‑інтерфейсів для скрінрідерів. |
| **Backlog** | Список завдань чи функцій продукту, що очікують реалізації. |
| **Benchmark** | Вимірювання продуктивності чи ефективності алгоритму/системи. |
| **Bias (упередженість)** | Систематичні похибки чи спотворення результатів, зумовлені даними чи алгоритмами. |
| **Bloom’s Taxonomy (Таксономія Блума)** | Модель навчальних цілей: пам’ять → розуміння → застосування → аналіз → оцінювання → створення. |
| **Boilerplate** | Шаблонний код або заготовка для пришвидшення розробки. |
| **C4 Model** | Ієрархія діаграм архітектури (Context–Container–Component–Code) для візуалізації систем. |
| **CI hook** | Автоматичний тригер/скрипт у конвеєрі CI, що запускає перевірки/збір метрик. |
| **CI/CD (Continuous Integration / Continuous Deployment)** | Процеси безперервної інтеграції та розгортання програмного забезпечення. |
| **CHANGELOG** | Журнал змін у проєкті, який містить список усіх нововведень, виправлень і оновлень версій. |
| **Code review** | Перевірка коду іншими учасниками команди для підвищення якості. |
| **Commit hash** | Унікальний ідентифікатор версії коду у системі контролю версій. |
| **CONTRIBUTORS.md** | Файл у репозиторії, що містить список учасників проєкту та їхній внесок. |
| **Copilot (GitHub Copilot)** | AI code assistant для автодоповнення, рефакторингу та пояснення коду у редакторі (VS Code тощо). |
| **Coverage (test coverage)** | Частка коду, покрита автоматизованими тестами. |
| **CVE (Common Vulnerabilities and Exposures)** | Публічний реєстр ідентифікаторів відомих вразливостей. |
| **Dashboard (дашборд)** | Інтерактивна панель показників/графіків для моніторингу метрик. |
| **Dockerfile** | Декларативний файл інструкцій для побудови Docker‑образу. |
| **Edge cases** | Крайові випадки або нетипові сценарії, які потрібно враховувати при тестуванні. |
| **ER‑модель / ERD (Entity‑Relationship Diagram)** | Діаграма «сутність‑зв’язок» для проєктування баз даних. |
| **EXPLAIN / ANALYZE (SQL)** | Інструменти СУБД для перегляду плану виконання та вимірювання продуктивності запитів. |
| **Few-shot** | Метод, коли моделі ШІ дають кілька прикладів для налаштування відповіді. |
| **Flexbox** | Модель компонування в CSS для гнучкого розташування елементів у рядках/стовпцях. |
| **Fork** | Копія репозиторію, створена для незалежної роботи над проєктом, яка дозволяє вносити зміни, не впливаючи на оригінальний репозиторій. |
| **Git** | Розподілена система контролю версій, яка дозволяє відстежувати зміни в коді та співпрацювати над проєктом. |
| **GitHub, GitLab** | Популярні онлайн‑платформи для розміщення репозиторіїв, співпраці над кодом, ведення задач і CI/CD. |
| **GPL, MIT, Apache-2.0** | Поширені типи ліцензій для програмного забезпечення з різними умовами використання. |
| **Integration tests (інтеграційні тести)** | Перевіряють взаємодію кількох модулів/сервісів у сукупності та їхню сумісність. |
| **Issue** | Опис проблеми, багу або завдання у системі трекінгу проєкту (наприклад, GitHub Issues, Jira). |
| **KPI (Key Performance Indicator)** | Ключовий показник ефективності; метрика для вимірювання досягнення цілей. |
| **Lighthouse** | Інструмент Chrome для аудиту веб‑сторінок (продуктивність, a11y, SEO, best practices). |
| **LLM (Large Language Model)** | Велика мовна модель, яка генерує текст на основі навчання на великих корпусах даних. |
| **Live-coding** | Написання чи модифікація коду в режимі реального часу (наприклад, під час захисту роботи). |
| **Lock‑file** | Файл фіксації версій залежностей для відтворюваних збірок (напр., `package‑lock.json`, `poetry.lock`). |
| **Looker Studio (ex-Data Studio)** | Сервіс Google для побудови дашбордів і візуалізації даних. |
| **Media queries** | CSS‑правила для адаптивної верстки під різні ширини/пристрої. |
| **Mermaid (Mermaid.js)** | Текстовий синтаксис для генерації діаграм (UML, C4, послідовності тощо) у Markdown. |
| **Merge request / Pull request** | Запит на внесення змін із однієї гілки або форку до основного репозиторію; проходить перевірку та обговорення перед злиттям. |
| **Mobile‑first** | Підхід до адаптивної верстки, коли дизайн і CSS спершу створюються для мобільних екранів. |
| **NDA (Non‑Disclosure Agreement)** | Угода про нерозголошення конфіденційної інформації. |
| **Normalization (1НФ–3НФ)** | Процес упорядкування схеми БД для зменшення надмірностей і аномалій. |
| **PDCA (Plan‑Do‑Check‑Act)** | Цикл безперервного вдосконалення «сплануй‑виконай‑перевір‑дій». |
| **Peer-review** | Взаємна експертна оцінка або рецензування робіт студентами чи колегами з метою підвищення якості, об’єктивності та розвитку навичок критичного аналізу. |
| **Pattern Repository (патерн «Репозиторій»)** | Шар доступу до даних, який ізолює доменну логіку від джерела даних. |
| **Production‑ready** | Готовність рішення до використання в продакшені: логування, обробка помилок, належне тестове покриття, конфігурація, безпека, CI/CD та документація. |
| **prompts.md** | Артефакт із промптами та ланцюжками промптів, що використовувалися у проєкті (трасування взаємодії з ШІ). |
| **Prompt / Prompt chaining** | Запит до ШІ / послідовність запитів, що дають змогу отримати складніший результат. |
| **Prompt engineering (промпт‑інжиніринг)** | Практика створення, формулювання та оптимізації запитів (промптів) до моделей ШІ для отримання максимально релевантних і якісних результатів. Включає техніки уточнення інструкцій, приклади few‑shot, обмеження та структурування контексту. |
| **Property‑based tests (тести на основі властивостей)** | Автоматично генерують різноманітні вхідні дані та перевіряють інваріанти/властивості системи (напр., Hypothesis, QuickCheck). |
| **Pull** | Операція отримання останніх змін із віддаленого репозиторію у локальний. |
| **Push** | Операція відправлення локальних змін у віддалений репозиторій. |
| **README.md** | Основний файл із описом проєкту, інструкціями щодо використання, встановлення та іншою важливою інформацією. |
| **Random seed (seed)** | Початкове значення для генераторів випадкових чисел; потрібне для відтворюваності експериментів. |
| **Refactoring (рефакторинг)** | Перепроектування або оптимізація коду без зміни його функціоналу. |
| **Repository (репозиторій)** | Сховище коду та матеріалів проєкту (наприклад, GitHub, GitLab). |
| **Reproducibility** | Відтворюваність результатів за тих самих умов (дані, код, інструкції). |
| **SBOM (Software Bill of Materials)** | Перелік компонентів та залежностей програмного продукту. |
| **SAST / DAST** | Static/Dynamic Application Security Testing — статичне/динамічне тестування безпеки. |
| **Scaffolding tool** | «Інтелектуальна опора», допоміжний інструмент для підтримки процесу навчання. |
| **Secret‑scan** | Перевірка репозиторію на наявність секретів/ключів доступу. |
| **SOLID / SRP (Single Responsibility Principle)** | Набір принципів ООП‑проєктування; SRP — кожен клас має одну відповідальність. |
| **SQL (Structured Query Language)** | Мова структурованих запитів для роботи з базами даних. |
| **SQL DDL / DML / DQL** | Підмножини SQL: визначення даних (CREATE/ALTER), маніпулювання (INSERT/UPDATE/DELETE) та запити (SELECT). |
| **Static analysis (статичний аналіз)** | Аналіз коду без виконання (lint, типи, уразливості). |
| **Stylometry (стилометрія)** | Аналіз «почерку» коду/тексту за стилістичними ознаками. |
| **Synthetic data** | Штучно згенеровані дані для тестування. |
| **Tag** | Мітка у системі контролю версій, що позначає певний реліз або важливий стан репозиторію. |
| **TTS (Text‑to‑Speech)** | Технологія перетворення тексту на синтезовану мову. |
| **Trade‑offs** | Компроміси між альтернативами (напр., швидкість vs. пам’ять, зручність vs. безпека); обґрунтований вибір із фіксацією наслідків у ADR. |
| **TypeScript** | Надбудова над JavaScript із системою типів. |
| **UI (User Interface)** | Користувацький інтерфейс продукту. |
| **UML (Unified Modeling Language)** | Уніфікована мова моделювання для візуалізації та проєктування систем. |
| **Unit tests (юніт‑тести)** | Перевіряють невеликі одиниці коду (функції/класи) ізоляційно від зовнішніх залежностей. |
| **Usability testing (юзабіліті‑тестування)** | Перевірка з користувачами: успішність виконання завдань, час, помилки. |
| **User stories** | Короткі сценарії, що описують потреби користувача у вигляді функціональних вимог. |
| **UX (User Experience)** | Досвід користувача під час взаємодії з продуктом. |
| **Валідація HTML (W3C Validator)** | Перевірка відповідності HTML‑коду стандартам W3C. |
| **ШІ (AI, Artificial Intelligence)** | Штучний інтелект, система, здатна виконувати завдання, які традиційно потребують людського інтелекту. |
| **Шкала Лайкерта (Likert scale)** | Оцінювання тверджень за фіксованою шкалою (напр., 1–5) в опитуваннях. |
